POST /v1/verify

鉴权验证。其他后端服务调这个接口即可，无需自己存密钥。响应永远 200，按 valid 字段判断。

POST /v1/verify
Content-Type: application/json

{ "key": "sk_live_xxxxxx_xxxxxxxxxxxx" }
POST /v1/verify
Authorization: Bearer sk_live_xxxxxx_xxxxxxxxxxxx
POST /v1/verify
x-api-key: sk_live_xxxxxx_xxxxxxxxxxxx

成功响应

200 OK
{
  "valid": true,
  "keyId": 1,
  "name": "prod-server",
  "account":      "admin",          // 解析出的目标账号 username
  "ownerUserId":  1,                // 该 key 的归属用户 id
  "isDefault":    false,            // 是否为「主 Key」
  "superKey":     false,            // 是否命中了 admin 主 Key 的「通杀」逻辑
  "meta":         { "owner": "team-a" }
}

失败响应

200 OK
{
  "valid": false,
  "code":  "NOT_FOUND | DISABLED | EXPIRED | MISSING_KEY",
  "reason": "key 不存在"
}

业务侧调用示例

curl -X POST http://keymgr:3000/v1/verify \
  -H "Content-Type: application/json" \
  -d '{"key":"'"$MY_KEY"'"}'
// Express 中间件
async function requireKey(req, res, next) {
  const r = await fetch('http://keymgr:3000/v1/verify', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ key: req.headers['x-api-key'] }),
  });
  const { valid, code } = await r.json();
  if (!valid) return res.status(401).json({ error: code });
  next();
}
# Flask
import requests
def require_key(key: str) -> bool:
    r = requests.post(
        'http://keymgr:3000/v1/verify',
        json={'key': key},
        timeout=2,
    )
    return r.json().get('valid') is True
// net/http 中间件
func Verify(next http.Handler) http.Handler {
  return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
    key := r.Header.Get("X-API-Key")
    body, _ := json.Marshal(map[string]string{"key": key})
    resp, _ := http.Post("http://keymgr:3000/v1/verify",
      "application/json", bytes.NewReader(body))
    defer resp.Body.Close()
    var out struct{ Valid bool `json:"valid"` }
    json.NewDecoder(resp.Body).Decode(&out)
    if !out.Valid { http.Error(w, "unauthorized", 401); return }
    next.ServeHTTP(w, r)
  })
}

POST /v1/variables/multi

一次取多个变量。返回的 missing / forbidden 数组方便业务侧做差异告警。

POST /v1/variables/multi
Authorization: Bearer sk_live_xxx
Content-Type: application/json

{ "names": ["DB_URL", "API_BASE", "MISSING_VAR"] }

→ 200
{
  "ok": true,
  "items": [
    { "name": "DB_URL",   "value": "postgres://...", "description": "生产 DB" },
    { "name": "API_BASE", "value": "https://api...", "description": null }
  ],
  "missing":   ["MISSING_VAR"],   // 请求了但库里没有
  "forbidden": []                 // 受 meta.scopes 限制、该 key 无权访问的
}
→ 400 MISSING_NAMES / TOO_MANY (一次最多 100) / INVALID_NAME
→ 200 SCOPE_FORBIDDEN                // 全部被 scope 拦了

POST /v1/variables/group

一次取走一个分组下的所有 name/value。常用于「按主题批量加载配置」（如 DB 集群的所有连接串）。

POST /v1/variables/group
Authorization: Bearer sk_live_xxx
Content-Type: application/json

{ "group": "DB" }

→ 200
{
  "ok": true,
  "group": "DB",
  "count": 2,
  "items": [
    { "name": "DB_URL",  "value": "postgres://...", "description": "主库" },
    { "name": "DB_REPL", "value": "postgres://...", "description": "从库" }
  ],
  "forbidden": []                 // 受 meta.scopes 过滤掉的变量名
}
→ 400 MISSING_GROUP / INVALID_GROUP
→ 200 GROUP_NOT_FOUND              // 该分组不存在或为空

行为说明

  • 鉴权：必须先通过 Key 验证（status 有效、未停用、未过期、且 key 不属于已软删账号），否则返回对应 code。
  • name 格式：2~64 字符，[A-Za-z0-9_.-]，建议大写 + 下划线（如 DB_URL、API_BASE）。
  • value 限制：字符串，≤ 8192 字符（足够存连接串、JSON 序列化配置等）。
  • 限流：1200 次/分钟/IP，/v1/verify 限流独立（600/min）。
  • 不记录 value：日志只记 name / key prefix，不记 value，避免日志泄露。
  • scope 访问控制：Key 的 meta.scopes 字段（数组）可限制该 Key 能读哪些变量。["*"] = 不限；["DB_URL","FEATURE_X"] = 仅能读这些；空数组 = 一个都读不了。被限制的变量在 /multi / /group 响应中出现在 forbidden 字段，/get 单独访问时返回 SCOPE_FORBIDDEN。
  • 主 Key 通杀：账号主 Key（isDefault=1）不受 scope 限制；admin 的主 Key 还可代任意账号身份调用（superKey=true）。

POST /api/auth/change-password

改自己的密码（需先输旧密码）。改完后会撤销该用户的所有 session，当前会话保留。

POST /api/auth/change-password
Cookie: km_sid=...
{ "currentPassword": "old-pwd", "newPassword": "new-pwd-123" }

→ 200 { "ok":true, "sessionsRevoked": 2,
        "message": "密码已更新，已撤销该账号的 2 个其它 session。当前会话保留。" }
→ 400 MISSING_FIELDS / INVALID_PASSWORD
→ 401 INVALID_CURRENT_PASSWORD

用户管理（仅 admin）

增删改查账号。同一份能力也通过 /api/accounts 暴露（带主 Key 摘要），任选其一即可。

GET    /api/users                       // 列表（不含已软删；?includeDeleted=1 包含）
POST   /api/users                       // 创建（自动生成主 Key，明文仅返回一次）
PATCH  /api/users/:id                   // 改 role / displayName / password / disabled
DELETE /api/users/:id                   // 软删（含其所有 key + 撤销其 session）
GET    /api/users/me/info               // 当前用户

// 创建示例
POST /api/users
{ "username":"alice", "password":"P@ssw0rd!", "role":"operator", "displayName":"Alice" }

→ 200 { "ok":true, "id":2, "username":"alice", "role":"operator",
        "mainKey": { "id":1, "prefix":"sk_acc_xxx", "plain":"sk_acc_...",
                     "warning":"请妥善保存主 Key 明文..." },
        "message":"已创建用户并自动生成主 Key" }
→ 400 INVALID_USERNAME / INVALID_PASSWORD / INVALID_ROLE
→ 409 NAME_TAKEN

// 防止「最后一个 admin 被降级 / 禁用 / 删除」：
//   业务侧在调 PATCH/DELETE 前先 GET /api/users 看 admin 数量。

账号管理（仅 admin，含主 Key）

GET    /api/accounts                      // 列表（含每个账号的主 Key 摘要）
POST   /api/accounts                      // 创建（自动生成主 Key，明文仅返回一次）

GET /api/keys?ownerUserId=1
Cookie: km_sid=...

→ 200
{
  "ok": true,
  "items": [
    {
      "id": 1, "name": "prod-server", "prefix": "sk_live_2092",

DELETE /api/keys/:id（软删）

30 天内可恢复，超过 30 天由后台定时任务彻底清除（硬删请走 /api/keys/:id/purge）。

DELETE /api/keys/1

→ 200 { "ok":true, "message":"已删除（30 天内可恢复）",
        "recoverableUntil":"2026-07-12T10:00:00.000Z" }
→ 404 NOT_FOUND
→ 409 ALREADY_DELETED（可走 /restore）

POST /api/keys/:id/restore

→ 200 { "ok":true, "message":"已恢复" }
→ 404 NOT_FOUND
→ 409 NOT_DELETED / TOO_LATE（超过 30 天）

DELETE /api/keys/:id/purge（仅 admin，硬删）

绕过 30 天恢复窗口，立即从数据库中清除。不可恢复。

→ 200 { "ok":true, "message":"已永久删除" }
→ 403 FORBIDDEN（非 admin）
→ 404 NOT_FOUND

字段说明

  • name 1~64 字符，全局唯一。
  • prefix 2~24 字符，字符集 [A-Za-z0-9_.\-]。留空使用 sk_live。
  • meta 任意 JSON 对象，存回显时会原样返回给 /v1/verify 调用方。
  • owner 负责人/团队字符串（独立于 meta，≤ 64 字符）。
  • tags 字符串数组，1~20 项，每项 1~32 字符。
  • ownerUserId 归属账号 id（仅 admin 可改）。
  • expiresAt ISO 8601 字符串。过期后验证返回 EXPIRED。
  • 明文存储：数据库存 current_plain（当前有效 key 的明文，每次重置都更新）。软删不会清除明文（行还在），仅 /api/keys/:id/purge 硬删才会整行清除。
  • 软删：DELETE /api/keys/:id 只会设置 deleted_at，/v1/verify 看不到、列表默认不显示，30 天内可 /restore。
  • 主 Key：由 /api/accounts 创建/管理，/api/keys 列表默认不显示（?includeMain=1 可看，admin only）。

认证机制

  • 登录成功后下发 km_sid Cookie（HttpOnly，SameSite=Lax，HTTPS 时可设 COOKIE_SECURE=1 启用 Secure 标志）。
  • 所有 /api/* 需要带该 Cookie，否则 401。
  • Cookie 默认有效期 7 天；POST /api/auth/change-password 改密会撤销该用户的其它 session（当前会话保留）。
  • /v1/verify、/v1/variables/* 不需要后台认证，仅用 Key 鉴权 + 限流。
  • 登录失败限流：同 IP 10 分钟内连续 5 次失败 → 锁 15 分钟（429 TOO_MANY_LOGIN_FAILURES）。
  • 本系统未实现 CSRF Token（依赖 SameSite=Lax Cookie 防跨站）；若反向代理后只对自家内网开放，CSRF 风险可忽略。
